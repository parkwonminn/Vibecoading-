# GPT.md — SnapCal

## 프로젝트 한 줄 정의
- **SnapCal**은 음식 사진 한 장으로 음식명과 예상 칼로리를 빠르게 보여주고, 그 결과를 기기 안에 기록하는 **로컬 우선 칼로리 기록 앱**이다.

## 최종 방향성
- **제품 이름:** `SnapCal` 유지
- **핵심 가치:** 빠름, 단순함, 프라이버시, 낮은 학습 비용
- **제품 전략:** `No Login`, `Local First`, `One-Tap Capture`, `Minimal UI`
- **우선순위:** 복잡한 기능 추가보다 **촬영 → 분석 → 저장 → 조회** 흐름을 가장 매끄럽게 완성한다.

## 제품 원칙
- 앱을 켜면 사용자는 오래 생각하지 않아야 한다.
- 로그인, 회원가입, 피드, 소셜 기능은 넣지 않는다.
- 음식 사진이 가장 중요한 콘텐츠이므로 UI는 사진을 방해하지 않아야 한다.
- 사용자는 칼로리의 **정확한 의학 수치**보다 **빠른 대략값**을 원한다.
- 저장된 기록은 언제든 다시 볼 수 있어야 하며, 탐색 비용이 낮아야 한다.

## 사용자 흐름
1. 앱 실행 즉시 메인 분석 화면이 열린다.
2. 사용자는 카메라 촬영 또는 갤러리 선택 중 하나를 고른다.
3. 선택된 음식 이미지를 분석한다.
4. 화면에는 음식명, 예상 칼로리, 분석 신뢰 문구가 표시된다.
5. 사용자가 저장하면 로컬 히스토리에 기록된다.
6. 사용자는 날짜순 히스토리에서 과거 분석 결과를 다시 확인할 수 있다.

## MVP 범위
### 반드시 포함
- 카메라로 음식 사진 촬영
- 갤러리에서 음식 사진 선택
- 음식명과 예상 칼로리 표시
- 분석 결과 로컬 저장
- 날짜별 또는 최신순 히스토리 조회

### 가능하면 포함
- 재분석 버튼
- 저장 전 취소 버튼
- 오늘 섭취 총 칼로리 합계

### 제외
- 로그인/회원가입
- 서버 기반 사용자 계정
- 커뮤니티 기능
- 결제 기능
- 복잡한 영양소 분석

## 기술 스택 결정
- **프레임워크:** React Native + Expo
- **언어:** TypeScript
- **이미지 입력:** Expo Camera, Expo Image Picker
- **로컬 저장:** 초기 MVP는 `AsyncStorage`로 시작하되, 기록 구조가 늘어나면 `expo-sqlite`로 확장할 수 있게 저장 계층을 분리한다.
- **AI 분석:** `FoodAnalysisService` 인터페이스를 기준으로 구현한다.

## AI 전략
### MVP 전략
- 첫 출시 속도와 결과 품질을 위해 **API 기반 이미지 분석**을 우선 허용한다.
- 단, 앱의 저장 데이터는 로컬에만 유지하고, 분석 요청 외의 사용자 데이터는 서버에 저장하지 않는다.
- 분석 응답은 반드시 다음 구조로 정규화한다.

```ts
type FoodAnalysisResult = {
  detectedFoodName: string;
  estimatedCaloriesKcal: number;
  analysisSummary: string;
};
```

### 장기 전략
- 추후 `OnDeviceFoodClassifier`를 추가할 수 있도록 AI 계층을 분리한다.
- UI 레이어는 어떤 분석 엔진을 쓰는지 몰라도 동작해야 한다.

## 데이터 구조
### 권장 저장 모델
```ts
type CalorieHistoryRecord = {
  historyRecordId: string;
  localImageUri: string;
  detectedFoodName: string;
  estimatedCaloriesKcal: number;
  analysisSummary: string;
  createdAtIsoString: string;
};
```

### 저장 원칙
- 이미지 원본 경로와 분석 결과를 함께 저장한다.
- 날짜 정렬이 가능하도록 시간은 ISO 문자열로 저장한다.
- 삭제와 재렌더링이 쉬운 고유 ID를 사용한다.

## 화면 구성 원칙
### 1. 메인 분석 화면
- 가장 먼저 보여야 하는 것은 카메라 또는 이미지 선택 진입점이다.
- CTA는 적고 강해야 한다.
- 분석 전과 후의 상태 차이가 분명해야 한다.

### 2. 분석 결과 화면
- 음식명
- 예상 칼로리
- 간단한 설명 문구
- 저장 버튼
- 다시 촬영 버튼

### 3. 히스토리 화면
- 최신 기록이 위에 오도록 정렬한다.
- 한 카드 안에 이미지, 음식명, 칼로리, 날짜를 보여준다.
- 빈 상태 화면은 친절하고 간단해야 한다.

## 디자인 시스템
- **배경:** `#FFFFFF`, `#F5F5F5`
- **텍스트 기본:** `#1A1A1A`
- **보조 텍스트:** `#757575`
- **포인트 컬러:** `#FF6B35`
- **원칙:** 포인트 컬러는 셔터 버튼, 주요 CTA, 칼로리 수치에만 사용한다.
- **컴포넌트 스타일:** 미니멀 카드 UI, 넉넉한 여백, 낮은 시각적 소음

## UX 세부 원칙
- 로딩은 짧고 명확해야 한다.
- 사용자는 지금 무슨 일이 일어나는지 항상 이해할 수 있어야 한다.
- 분석 실패 시에는 침묵하지 말고 다시 시도할 수 있게 안내한다.
- 칼로리는 `약 450 kcal`처럼 부담 없는 문장으로 노출한다.
- 의료 정보처럼 보이게 과장하지 않는다.

## 코드 작성 원칙
- 모든 함수, 클래스, 변수 이름은 **역할과 이유가 이름만으로 드러나야 한다.**
- `x`, `data`, `temp`, `handleThing` 같은 추상적인 이름은 사용하지 않는다.
- 여러 인자를 받을 때는 **객체 구조 분해 할당**을 우선 사용한다.
- 하나의 함수는 하나의 책임만 가진다.
- 복잡한 흐름은 작은 함수로 나누고, 상위 함수는 읽기만 해도 동작 흐름이 보여야 한다.
- UI 컴포넌트는 상태 표현과 이벤트 의도가 이름에서 드러나야 한다.

## 권장 모듈 구조
```txt
app/
  screens/
  components/
  hooks/
  services/
  storage/
  constants/
  types/
```

### 권장 책임 분리
- `services/food-analysis/`: 이미지 분석 요청과 응답 정규화
- `storage/calorie-history/`: 저장, 조회, 삭제
- `screens/`: 사용자 흐름 조합
- `components/`: 재사용 가능한 UI 조각
- `types/`: 공통 데이터 타입

## 완료 기준
- 사용자가 카메라 또는 갤러리에서 음식 사진을 입력할 수 있다.
- 앱이 음식명과 예상 칼로리를 사용자에게 이해하기 쉬운 형태로 보여준다.
- 사용자가 분석 결과를 로컬에 저장할 수 있다.
- 저장된 결과를 히스토리 화면에서 다시 볼 수 있다.
- 전체 UI는 무채색 기반이며 포인트 컬러 하나만 사용한다.
- 코드 구조가 기능 단위로 분리되어 있어 다음 단계 확장이 가능하다.

## 구현 시 주의사항
- 지금 단계에서는 기능 수를 늘리는 것보다 핵심 흐름의 완성도를 높인다.
- 프라이버시 관련 문구는 과장하지 말고 실제 동작과 일치해야 한다.
- AI 결과는 항상 부정확할 수 있으므로 실패 상태와 예외 처리를 반드시 포함한다.
- 저장소 구현은 나중에 교체 가능하도록 `StorageAdapter` 형태로 추상화하는 것이 좋다.
